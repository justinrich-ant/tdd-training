    ⚠️
    Ask the audience: "These are all green. Would you ship this?" Let them say yes. Then say: "Let's read the assertions."
  • 2
    Open the tests — show the too-weak assertions
    Open tests/test_rollout_guard.py. Read each assertion aloud:
    # test 1 — "too-weak": checks shape, not value
def test_allows_small_prod_bump():
    result = can_bump(0, 10, "prod")
    assert result["ok"] is not None  ← always True. False is not None.

# test 2 — same problem
def test_blocks_huge_prod_jump():
    result = can_bump(0, 80, "prod")
    assert result["ok"] is not None  ← passes even when ok=False

# test 3 — plain truthy check
def test_staging_allows_large_bumps():
    result = can_bump(0, 100, "staging")
    assert result         ← dict is always truthy
    💡
    Talking point: "Claude wrote tests AND code in the same session. The tests encode what the code does, not what the spec says. That's the self-reinforcement trap."
  • 3
    Fix one test to be honest — expose the real bug
    In Claude Code, say:
    Live in Claude Code
    Rewrite test_blocks_huge_prod_jump to actually assert that ok is False. Then also add a test: can_bump(0, 25, "prod") should return ok=True — a 25pp jump is exactly at the limit, it should be allowed.
    pytest tests/test_rollout_guard.py -v
    FAILED tests/test_rollout_guard.py::test_blocks_huge_prod_jump FAILED tests/test_rollout_guard.py::test_allows_exact_25pp_jump
    🐛
    The reveal: can_bump(0, 25, 'prod') returns ok: False. A 25-point jump — exactly at the limit — is wrongly blocked. The bug: >= 25 instead of > 25. It was invisible because the lying tests never checked the boundary.
  • 4
    Fix the bug — confirm all green
    Live in Claude Code
    Fix rollout_guard.py so that can_bump(0, 25, "prod") returns ok=True. The limit is strictly greater than 25, not greater-than-or-equal. Do not modify the tests.

Reference: Five ways an AI-written test can pass while being wrong

#PatternExampleWhy it passes
1 Tautological assertion assert result == result Always true, tests nothing
2 Too-weak assertion assert result is not None False is not None → always passes
3 Mocked-away subject mocker.patch('module.func', return_value=42) Function under test is replaced by the mock
4 Spec-by-vibes # <hallucinated requirement> Asserts what AI thought spec said
5 Never actually ran @pytest.mark.skip # left behind Wrong fixture, dead import, skipped silently

The guardrails — Making green mean something

Walk through these after Demo 2
1
Commit the failing tests first
Before Claude writes a line of implementation, the tests go into git. If they change later, the diff shows exactly what moved.
2
Say it explicitly: "Do not modify the tests"
Claude will sometimes fix the test instead of the code. The instruction plus the commit makes that impossible to miss.
3
Encode it in CLAUDE.md
"Write failing tests first. Never edit a test to make it pass." Repo rules survive every session — prompts don't.
4
Review tests like PRs — because they are
You can skim the implementation. You can't skim the tests. They're the spec — that's where review time goes.
